PALANTIR PRODUCT CLONING SPECIFICATION: THE TITANIUM STANDARD
============================================================
Strategic Goal: This guide provides the definitive blueprint for reformatting 
any application or website into a Palantir-tier intelligence tool. It mirrors 
the visual DNA and structural logic of products like Gotham, Foundry, and Gaia.

1. THE PHILOSOPHICAL ORIGINS (THE "WHY")
----------------------------------------
Palantir's clinical look is not an artistic choice; it is an architectural 
manifestation of these core technical bibles:

* DOMAIN-DRIVEN DESIGN (Eric Evans)
  - WHY: Palantir deals with messy real-world problems like war, fraud, and 
    disease. They needed a way to make software that thinks like an expert.
  - THE RESULT: The "Object-Centric" model. In their apps, you don't see 
    spreadsheets; you see "Objects" (a person, a car, a phone). Because the 
    code treats these as real things, the UI looks like a digital twin of 
    reality.

* CLEAN ARCHITECTURE (Robert C. Martin)
  - WHY: They wanted software that could last 20 years and work with any 
    hardware (any satellite, any camera, any database).
  - THE RESULT: The "Lens" effect. The UI is designed to be a very thin, 
    transparent window into the data. This is why it is so monochrome and 
    simple—it's meant to get out of the way so the data can speak.

* EFFECTIVE JAVA (Joshua Bloch)
  - WHY: To ensure the system never crashes when lives are on the line. 
    They use rigid, "bulletproof" coding patterns.
  - THE RESULT: The "Heavy" feel. Every button and panel feels intentional 
    and solid. Because the data is "immutable" (it can't be accidentally 
    changed), the UI feels like a permanent record of truth.

* A PHILOSOPHY OF SOFTWARE DESIGN (John Ousterhout)
  - WHY: To manage "Information Overload."
  - THE RESULT: "Deep Modules." Their tools look simple on the outside (a 
    clean sidebar or a small map icon), but when you click them, they reveal 
    massive power. They hide complexity until the user actually needs it.

* DESIGNING DATA-INTENSIVE APPLICATIONS (Martin Kleppmann)
  - WHY: To handle billions of signals in real-time without the app lagging.
  - THE RESULT: The 4D Timeline. Because they architected their data around 
    "Events," their products (like Gaia) allow you to slide a bar and watch 
    the entire world move backward or forward in time.

* IMPRO: IMPROVISATION AND THE THEATRE (Keith Johnstone)
  - WHY: To teach engineers to handle the "Forward Deployed" chaos of 
    working in a war zone or a disaster site.
  - THE RESULT: "Adaptive" UIs. If a sensor goes offline, the UI doesn't 
    break; it "improvises" and shifts focus to whatever data is still 
    available. It feels alive and reactive.

2. THE VISUAL DNA (THE "LOOK")
------------------------------
Because of the books above, a Palantir product always follows these rules:

* THE PALETTE (OKLCH Color Space):
  - Primary Background: Deep Charcoal [oklch(20% 0.02 260)] - Like a dark 
    control room.
  - Surface/Panels: Elevated Charcoal [oklch(25% 0.02 260)] - To show 
    hierarchy without using bright colors.
  - Primary Text: Pure White [oklch(98% 0 0)] - For maximum readability.
  - Intent Colors: Neon Blue (Logic), Neon Green (Friendly), Amber (Warning), 
    Red (Hostile). Colors are ONLY used to convey meaning, never for style.

* THE GEOMETRY:
  - Border Radius: Strictly 0px to 2px. No rounded corners. Circles are 
    for units on a map; squares are for the interface.
  - Borders: 1px solid lines. Everything is divided by thin "Logic Lines."
  - Padding: Compact. Space is a resource; don't waste it on "fluff."

3. THE UNIVERSAL REFORMATTING TEMPLATE
--------------------------------------
Apply these four layers to any interface to transform it into a Palantir tool:

LAYER 1: THE WORKSPACE SHELL (TOP NAV)
- Replace sidebars with a horizontal tab bar at the very top.
- Tabs = Workspaces (e.g., Object Explorer, Signal Map, Code Terminal).

LAYER 2: THE INFINITE CANVAS (CENTER)
- The main window is a spatial environment (3D Map or Node-Link Graph).
- Data is represented as discrete, clickable objects floating in 3D space.

LAYER 3: THE OBJECT DRAWERS (RIGHT SIDE)
- NO POP-UPS. When an object is clicked, a drawer slides out from the right.
- This drawer is the "Dossier"—it shows every detail about that specific object.

LAYER 4: THE MISSION LAYER (FLOATING)
- Use semi-transparent black overlays in the corners for pinned items, 
  high-priority alerts, and tactical chat.

4. THE CLONING PROTOCOL (4-STEP ACTION)
---------------------------------------
1. FLATTEN: Remove shadows, gradients, and all rounded corners.
2. DARKEN: Apply the OKLCH Charcoal palette.
3. DENSIFY: Shrink text to 12px and reduce whitespace by 50%.
4. ONTOLOGIZE: Turn every list item into a clickable object with a right-side drawer.
