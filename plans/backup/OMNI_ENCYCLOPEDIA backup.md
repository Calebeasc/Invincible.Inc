# 1. Executive summary

Omni should be built as a **native desktop mission application** with a dense but understandable operator interface, a strong domain model, a graph-first data backbone, and a modular service architecture. Its job is not merely to display data. Its job is to:

1. ingest many authorized data streams,
2. normalize and relate them,
3. expose them through a clear ontology,
4. surface confidence-scored insights and alerts,
5. support human workflows and decisions,
6. preserve audit, provenance, and operator accountability,
7. continue operating in constrained and degraded environments.

The product should feel like a **high-control, high-context operations console**, but it must remain original in branding, layout system, and interaction details. It may be inspired by the genre of command-and-control analytics platforms, but it should not copy trade dress, proprietary workflows, or branded product identity.

### Core reframed outcome
Omni is not a â€œgod toolâ€ in the sense of unconstrained power. It is a **governed operations platform** that gives the right people the right view of the right data, with the right approvals, at the right time.

# 2. Design foundations from the books

This manual applies the principles from the first study guide, especially:

## 2.1 Clean Architecture
Use Clean Architecture to separate:
- enterprise rules,
- use cases,
- adapters,
- infrastructure and delivery mechanisms.

That means:
- desktop UI is a detail,
- database choice is a detail,
- model provider choice is a detail,
- connector technology is a detail,
- the core mission workflows and domain language stay stable.

### Clean Architecture implications for Omni
- The domain model cannot depend on WPF, WinUI, Qt, Electron, or any UI toolkit.
- The domain model cannot depend on a specific graph database.
- The use cases cannot depend on a specific AI vendor.
- Data connectors cannot be allowed to leak raw vendor semantics into the domain.
- Offline mode should degrade through adapters, not by rewriting core logic.

## 2.2 Domain-Driven Design
DDD matters because Omni is fundamentally a **meaning-making** product. The hard part is not drawing maps or graphs. The hard part is expressing the domain clearly enough that the system can support real decisions.

That means Omni needs:
- a ubiquitous language,
- bounded contexts,
- aggregates and invariants,
- clear ownership of models,
- explicit translation between data source terms and product terms.

## 2.3 Effective Java and API discipline
Even if much of Omni is built in C# and C++, the Effective Java mindset still applies:
- simple and explicit APIs,
- immutability where possible,
- builder patterns for complex construction,
- careful contracts,
- defensive copying at trust boundaries,
- strong exception taxonomy,
- concurrency discipline,
- versioned serialization and upgrade safety.

### In plain terms
Omni should not become an untestable pile of adapters, widgets, and async side effects. It needs a disciplined internal platform.

Omni is a **desktop-first, mission-support analytics application** that helps authorized users understand dynamic situations across geography, time, entities, events, systems, and evidence.

## 3.1 What it should feel like
The product should feel like:
- a live operational workspace,
- a linked graph and map explorer,
- a case and workflow manager,
- an alert and recommendation engine,
- a resilient field-operable client.

## 3.2 What it should not feel like
It should not feel like:
- a generic dashboard product,
- a thin web wrapper pretending to be desktop-native,
- a collection of disconnected widgets,
- a copy of another companyâ€™s branding,

# 5. Product principles

1. **Human-in-the-loop by default**  
   The system recommends, ranks, and explains. A human approves consequential actions.

2. **Provenance over magic**  
   Every insight should show where it came from.

3. **Original UX, genre-informed design**  
   Dense operational UI is fine. Copying proprietary product identity is not.

4. **Graceful degradation**  
   The product must work in normal, degraded, and isolated environments.

5. **Least privilege everywhere**  
   Users, services, connectors, models, caches, and updates all operate with constrained permissions.

6. **Audit everything important**  
   Query, export, model invocation, case changes, approvals, and administrative actions all leave durable trails.

7. **Offline is a first-class mode, not a broken fallback**  
   The desktop client should still provide value when disconnected.

8. **Modularity before ambition**  
   Build separable workstreams that can be independently tested and integrated.

# 6. Ubiquitous language

Omni needs a shared vocabulary so product, design, engineering, security, and operators stop talking past each other.

## 6.1 Core terms
- **Source**: an external system or feed.
- **Connector**: code that ingests and normalizes data from a source.
- **Record**: a raw or lightly normalized datum.
- **Entity**: a durable domain object, such as a person, device, account, asset, organization, vehicle, or location.
- **Event**: something that happened at a time.
- **Observation**: a source-derived claim with provenance.
- **Case**: a bounded operational or investigative workspace.
- **Alert**: a system-generated notice requiring review.
- **Insight**: a derived relationship, pattern, explanation, or recommendation.
- **Action**: a governed workflow step initiated by a user.
- **Playbook**: a reusable sequence of review and response tasks.
- **Evidence**: a preserved record attached to a case.
- **Provenance**: the chain of origin and transformation.
- **Confidence**: a score attached to a claim or inference.
- **Snapshot**: a saved state of app version, configuration, or data cache.
- **Mode**: normal, degraded, or isolated operating state.

## 6.2 Why this matters
A source may say â€œsubscriber,â€ another may say â€œaccount,â€ another may say â€œprincipal.â€ Omni should not leak each raw term upward. The domain should expose a consistent language and keep vendor-specific translation at the edges.

# 15. Security and trust model

## 15.1 Security objectives
- authenticate strongly,
- authorize precisely,
- minimize secrets exposure,
- protect local caches,
- verify updates cryptographically,
- preserve integrity in degraded modes,
- maintain auditability.

## 15.2 Authentication
Support:
- SSO in normal mode,
- smartcard or hardware-backed auth where available,
- local sealed offline auth package for continuity mode,
- short-lived offline access tokens with strict expiry,
- explicit privilege elevation flow for sensitive actions.

## 15.3 Authorization
Use layered authorization:
- role-based access control for broad capability classes,
- attribute-based rules for case, data class, geography, and sensitivity,
- approval workflows for exports and destructive actions.

## 15.4 Break-glass design
In a severe continuity scenario, access should still be possible, but safely.

Requirements:
- pre-provisioned break-glass identities,
- dual control or approval policy where feasible,
- local cryptographic verification,
- mandatory reason entry,
- visible session banner,
- heavy audit logging,
- post-event review requirement.

Do **not** design break-glass as an easy bypass. Design it as a constrained resilience mechanism.

## 15.5 Local storage
- local encrypted cache,
- per-user protected key material,
- separate store for snapshots and rollback bundles,
- secure deletion for expired sensitive material when possible,
- offline package expiration and integrity checks.

# 16. Deployment modes

## 16.1 Mode 1: normal connected mode
Characteristics:
- central services available,
- source sync active,
- collaboration active,
- remote analytics and model inference available,
- update checks enabled.

## 16.2 Mode 2: degraded continuity mode
Characteristics:
- central services partially unavailable,
- one workstation or small server node may act as local service host,
- local graph index and local inference enabled,
- selected cached data available,
- collaboration reduced or local-only,
- update checks deferred or manual.

## 16.3 Mode 3: isolated/offline mode
Characteristics:
- no central backend assumed,
- desktop app operates on local cache, local files, and permitted public data fetched when connectivity exists,
- case work, timeline review, map analysis, document analysis, and local recommendations still function,
- no assumption of privileged remote capabilities.

## 16.4 Degradation strategy
The same use cases should degrade by:
- smaller data window,
- reduced collaboration,
- slower analytics,
- reduced model sophistication,
- explicit freshness warnings,
- cached ontology and rules.

The core workflows should remain recognizable.

# 17. Update, rollback, and tray service

## 17.1 Tray service responsibilities
- show sync and mode status,
- expose settings,
- manual check for updates,
- show update progress,
- allow rollback to prior snapshot,
- package diagnostic bundle for support.

## 17.2 Update model
1. client checks signed manifest,
2. compares version and compatibility,
3. downloads staged package,
4. verifies signature and checksum,
5. runs preflight validation,
6. installs into staged side-by-side location,
7. performs first-run health checks,
8. marks new version active if healthy,
9. preserves rollback pointer to previous stable version.

## 17.3 Rollback model
Rollback should:
- restore the last stable app version,
- preserve user workspace state where compatible,
- preserve local data caches when safe,
- produce a failure report bundle,
- defer telemetry upload until connectivity exists.

## 17.4 Checks and balances
- no unsigned update can install,
- no auto-update should activate if core health checks fail,
- rollback should be one click for authorized admins,
- incompatible local schema migrations should require explicit confirmation or staging.

# 27. Risks and failure modes

## 27.1 Product risk
Trying to be everything for everyone will kill coherence.

Mitigation:
- choose 2 to 3 primary mission areas first.

## 27.2 Architecture risk
Too much logic in the desktop UI will make testing and portability collapse.

Mitigation:
- keep use cases and domain model outside UI.

## 27.3 Data risk
Bad identity resolution creates false certainty.

Mitigation:
- confidence scoring, merge review queue, provenance visibility.

## 27.4 AI risk
Analysts may trust polished nonsense.

Mitigation:
- provenance requirement, confidence display, no opaque auto-action.

## 27.5 Security risk

## 🏗️ VOLUME IX: THE ZERO-BROWSER MANDATE (WEB DE-COUPLING)

### 9.1 Strategic Rationale
To prevent AI agents and automated systems from introducing browser-dependent vulnerabilities or 'web-only' logic, Invincible.Inc has mandated the complete removal of Omni access from the public web. This ensures that the platform remains a pure native Windows application, utilizing the full performance and security of local hardware primitives.

### 9.2 The 'Native Only' Enforcement
- **Platform:** Omni is exclusively a native Windows application (WinUI 3, C#, C++).
- **Public Access:** Prohibited. No public URLs or web-accessible dashboards for the command core.
- **Exception:** The primary advertisement and explainer website remains the only web-facing asset for visibility and distribution.

### 9.3 AI Build Constraints
- **Zero Browser Dependencies:** AI agents are forbidden from creating components that rely on browser engines (Electron, WebView2) for core functionality.
- **Pure Native Primitives:** All new modules (UTT, LE-GOLIATH, Spectral Eye) must be implemented using native system calls and high-performance C++ or WinUI controls.
- **Data Ingestion:** While the web is used for raw data ingestion (APIs, Sockets), the processing and visualization layers must remain 100% browser-independent.

### 9.4 Architectural Impact
This shift moves Omni from a 'Hybrid App' to a 'Sovereign Native Environment,' reinforcing the doomsday-scenario requirement where the system must function without reliance on remote web services or browser-based runtimes.
